# Work Order Visibility & Access Control

## Overview

Every query that touches the `work_orders` table passes through a single, centrally-managed visibility filter.  The filter is generated by `getWorkOrderVisibilityFilter(context)` in `lib/work-orders/visibility.ts` and returns a `Prisma.work_ordersWhereInput` scoped to the logged-in user's role, department, and assignments.

The same filter is applied to:

| Surface | File |
|---------|------|
| List page — KPI cards, tab counts, table rows | `app/(dashboard)/maintenance/work-orders/page.tsx` |
| Detail page | `app/(dashboard)/maintenance/work-orders/[id]/page.tsx` |
| Print page | `app/(dashboard)/maintenance/work-orders/[id]/print/page.tsx` |

**Visibility is enforced in the database query — not in the UI.**  A user who navigates directly to `/maintenance/work-orders/<id>` or `/maintenance/work-orders/<id>/print` for a record outside their scope receives HTTP 404 from `notFound()`.

## Visibility Rules per Role

### Super Admin / IT Admin / Viewer–Auditor
**Filter:** `{}` — unrestricted, sees every work order

These roles require full system access for administration, audit, and support.  IT Admin's available *workflow actions* (approve, reject, assign, close) are still governed by their `role_permissions` rows; visibility and action authority are separate concerns.

---

### CEO / Management
**Filter:** High-priority active WOs + purchase-pipeline WOs + CEO-approval-stage purchase requests

```
(priority IN ('High','Urgent') AND status NOT IN terminal)
OR status = 'Waiting for Purchase'
OR purchase_requests.some.status = 'Pending CEO Approval'
```

The CEO oversees critical asset downtime and high-value purchase approvals.

---

### Finance Manager
**Filter:** WOs that have entered the finance/purchase approval pipeline

```
status = 'Waiting for Purchase'
OR purchase_requests.some.status IN (
   'Pending Finance Approval', 'Pending CEO Approval',
   'Approved', 'Ordered', 'Received'
)
```

Finance only needs the cost side of the workflow.  Unrelated drafts and in-progress jobs are not shown.

---

### Purchase Officer
**Filter:** WOs that have an active (non-terminal) purchase request

```
status = 'Waiting for Purchase'
OR purchase_requests.some.status NOT IN ('Cancelled', 'Rejected', 'Draft')
```

---

### Store Keeper
**Filter:** Only the two stages where store action is actively needed

```
status IN ('Waiting for Parts', 'Parts Issued')
```

Store Keepers must not see unrelated drafts, submitted forms, or closed records.

---

### Technician
**Filter:** Only WOs where the user has a direct `work_order_assignments` row

```
work_order_assignments.some.technician_id = userId
```

Technicians see only the jobs they are personally assigned to.

---

### Maintenance Supervisor
**Filter:** WOs assigned to them as supervisor + supervisor-stage WOs in their department

```
assigned_supervisor_id = userId
OR (
  requested_by_department_id = deptId
  AND status IN ('Approved', 'Assigned', 'In Progress',
                 'Waiting for Parts', 'Waiting for Purchase',
                 'Parts Issued', 'Completed by Technician')
)
```

If no department is assigned, the supervisor sees all supervisor-stage WOs globally (so they can still be assigned).

---

### Maintenance Manager
**Filter:** Only work orders from their own department

```
requested_by_department_id = deptId
```

**Intentional restriction:** Global pending-approval visibility is NOT granted.  A Maintenance Manager approves work orders that were created within their own department — not every pending WO across the company.  The `requested_by_department_id` field is the closest available proxy for "routed to this department for approval" given that the schema has no separate `routed_department_id` column.

**No-department fallback:** `created_by = userId` (least privilege).  This condition means the manager's profile is misconfigured; an admin should assign a department.

---

### Maintenance Data Entry
**Filter:** Only work orders they personally created

```
created_by = userId
```

Data entry staff create and submit their own WOs.  They cannot view records created by colleagues, even within the same department.

---

### Department Requester
**Filter:** Only work orders they personally created

```
created_by = userId
```

A Department Requester sees only the requests they submitted themselves.  This includes the full lifecycle of those requests (Draft through Closed / CEO approval), because visibility is scoped by authorship, not by current status.  They do **not** see all work orders belonging to their department.

---

### Fallback (unknown role)
**Filter:** `{ created_by: userId }` — principle of least privilege.

---

## What Changed from the Initial Implementation

| Role | Before | After |
|---|---|---|
| Maintenance Manager (with dept) | `dept WOs OR global pending-approval statuses` | `dept WOs only` |
| Maintenance Manager (no dept) | `{}` — all records | `created_by = userId` — least privilege |
| Department Requester | `created_by OR dept WOs` | `created_by only` |

**Why these changes matter for a large company:**

- A Maintenance Manager in the Mechanical department must not see Electrical or Civil department pending approvals.  The previous OR arm exposed the full company queue whenever a manager had `work_orders.approve` permission.
- A Department Requester (e.g. an Operations Coordinator) should track only their own requests.  Showing all department WOs would expose maintenance history submitted by other staff in the same department.

## How Filters Compose with User-Applied Filters

The list page stacks filters as a Prisma `AND` array to prevent key conflicts when the visibility filter itself uses a field (e.g. `status`) that the user also filters by:

```typescript
const listConditions = [
  { deleted_at: null },
  visibilityFilter,      // role scope — applied first
  { status },            // user tab/dropdown filter (if set)
  { priority },          // user dropdown filter (if set)
  // ...other user filters
];
const where = { AND: listConditions };
```

User filters **narrow within** the visible scope — they can never widen it.  A Store Keeper filtering by `status = "Draft"` gets zero results, because `status IN ('Waiting for Parts', 'Parts Issued') AND status = 'Draft'` is impossible at the SQL level.

---

## KPI Cards and Summary Counts

All KPI cards and workflow tab counts use `visibilityOnlyWhere` — the same visibility filter combined with `deleted_at: null`, **without** any user-applied filters.  This keeps the KPI numbers representative of the user's full operational scope regardless of what they have typed in the search or filter bar.

---

## Department Dropdown Scoping

The department filter dropdown on the list page shows only departments relevant to the user's scope:

| Role | Departments shown |
|------|-------------------|
| `maintenance_data_entry` (with dept) | Own department only |
| All other roles | All active departments |

---

## IT Admin Action Restriction

IT Admin has **full visibility** (`{}`), identical to Super Admin.  Their available workflow actions (approve, reject, assign, close) are governed by `role_permissions` rows — the same mechanism used by every other role.  The `WorkflowActions` component already gates every action button by `hasPermission()`, so no additional code is needed.

---

## Adding a New Role

1. Add the slug to `types/database.ts` → `RoleSlug` union.
2. Seed the role and permission rows.
3. Add a `case` to `getWorkOrderVisibilityFilter()`.
4. Add a `case` to `getRoleDescription()`.

## Testing Guidance

### Testing Maintenance Manager visibility

1. Log in as a Maintenance Manager profile that has `department_id = <dept-A-uuid>`.
2. Open Work Orders — the list, KPI counts, and tab counts must show **only** WOs where `requested_by_department_id = dept-A`.
3. Try navigating directly to `/maintenance/work-orders/<id>` for a WO belonging to a different department — expect HTTP 404.
4. Log in as a second Maintenance Manager with `department_id = <dept-B-uuid>` — confirm they see a different subset with no overlap from dept-A records.
5. Log in as a Maintenance Manager with **no department set** — expect to see only WOs they personally created (the misconfiguration fallback).

### Testing Department Requester visibility

1. Log in as Department Requester user A (dept = Operations).
2. Create two work orders as user A.
3. Log in as Department Requester user B in the **same** department.
4. User B must see zero records from the list — they have not created any.
5. Navigate directly to `/maintenance/work-orders/<id>` for one of user A's WOs — user B receives HTTP 404.
6. Log back in as user A — both WOs must be visible regardless of current status (even if they have progressed to Waiting for Purchase or Closed).

## Security Notes

- Visibility is enforced at the Prisma/SQL layer — URL manipulation cannot bypass it.
- Out-of-scope records return HTTP 404, not 403 — this avoids confirming that the record exists.
- The `work_orders.print` permission gate remains active on the print page in addition to the visibility check — both must pass.
- Cost fields on the detail page are additionally gated by `can_view_costs` on the profile, independently of visibility.
